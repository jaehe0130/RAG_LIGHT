# 찰칵! 소비자 공정 Guard - 공정 소비 신호등 AI 서비스

## 🚀 프로젝트 개요
'찰칵! 소비자 공정 Guard'는 사용자가 약관 이미지나 온라인 광고 캡처본을 업로드하면, 다중 지식베이스(공정거래위원회 의결서, 한국소비자원 상담 DB) RAG 구조와 AI 환각 차단 기술을 결합하여 불공정 조항 및 기만 광고 여부를 **'3색 신호등(안전/주의/위험)'**으로 판정해 주는 공공 AI 서비스입니다. 
위험(RED) 판정 시 관공서(공정위, 소비자원) 제출을 위한 **신고서 초안까지 원스톱으로 자동 생성**합니다.

---

## 🏗️ 시스템 아키텍처 및 데이터 흐름
1. **업로드**: React 프론트엔드에서 약관/광고 이미지 파일 및 타입 전송
2. **[OCR 노드]**: 문서 특성에 맞는 고성능 텍스트 추출 및 전처리 수행
3. **[RAG 노드]**: 공정위 의결서 DB 및 소비자원 분쟁 DB에서 Qdrant 하이브리드 검색을 통해 유사 사례 병렬 추출
4. **[검증 노드]**: OpenAI API 및 1,000건의 QA 데이터셋을 통한 프롬프팅, 룰 기반(Rule-based) 수치 교차 검증으로 최종 신호등 색상 확정
5. **[신고 노드]**: 위험 등급 시 관공서 양식에 맞는 구제 신청서/신고서 초안 자동 작성
6. **서빙**: 가공된 최종 JSON 데이터를 프론트엔드로 전송하여 대시보드 시각화

## 👥 팀원별 역할 및 전담 파일 가이드

원활한 협업과 Git 충돌 방지를 위해 각 팀원은 **본인에게 할당된 전담 파일**만 수정합니다.

<br/>

### 👑 팀원 A (홍제희) : 백엔드 리드 및 아키텍처 조립
> **주 사용 언어**: `Python`  
> **전담 작업 공간**: `backend/main.py`, `backend/models.py`
* **FastAPI 라우팅**: 메인 엔드포인트 세팅 및 API 응답(JSON) 규격 정의
* **LangGraph 설계**: 노드 간 상태 공유(AgentState) 및 파이프라인 조립
* **에이전트 제어**: 멀티 에이전트 워크플로우 통제 및 관공서 양식 초안 자동 작성
* **🚨 지속적 관리 및 모니터링 체크리스트**:
  - **파이프라인 상태(State) 검증**: 모듈을 거칠 때마다 `AgentState` 데이터가 누락 없이 잘 전달되는지 파이프라인 전체 흐름을 감시합니다.
  - **API 규격 엄수 (Contract Check)**: 프론트엔드 화면이 깨지지 않도록 최종 반환 JSON 포맷의 구조를 기획안과 100% 일치시킵니다.
  - **통합 예외/에러 핸들링**: DB 연결 끊김, API 타임아웃 등 장애 상황을 캐치하여 우회 메시지를 내려주도록 방어 코드를 구축합니다.

  - **직관적 업로드**: 약관이나 광고 캡처본 업로드를 위한 드래그 앤 드롭 UI 구현

  - **📊 맞춤형 피해 사례 트렌드 시각화**: 백엔드가 전달하는 `statistics` JSON 데이터(유사 사례 건수, 분쟁 비율)를 바탕으로, 차트 라이브러리(Recharts 등)를 활용해 "최근 회원님과 유사한 피해 접수 00건" 등의 트렌드 위젯/그래프 렌더링
  - **🚨 원클릭 피해 구제 연계 (신고서)**: 판정 결과가 **RED(위험)**일 때만 활성화되는 기능. 백엔드가 내려주는 `report_form`(소비자원 신청서 또는 공정위 신고서 초안) 텍스트를 화면에 띄우고 **[문서 복사하기]** 또는 **[PDF 다운로드]** 버튼을 만들어 원스톱 신고 로직 구현
* **파이썬 OCR 파이프라인 개발 가이드**:
  - 최적의 OCR 엔진 연동 및 OpenCV를 활용한 해상도 향상/이진화 전처리 로직 구현
  - RAG와 LLM이 이해하기 쉽도록 텍스트 덩어리를 정제하여 반환

### 🔍 팀원 C (이예경) : 데이터베이스 검색(RAG) 엔진 구현
> **주 사용 언어**: `Python`  
> **전담 작업 공간**: `backend/modules/rag_search.py` (검색용), `backend/data/ingest_qdrant.py` (데이터 적재용)
* **Qdrant 초기 데이터 적재 (Ingestion)**:
  - 원본 JSON 파일(`raw/` 폴더)을 `intfloat/multilingual-e5-large-instruct` 임베딩 모델로 벡터화 후 Qdrant 로컬 DB에 적재
  - `ftc_decisions`(공정위 의결서), `kca_cases`(소비자원 피해구제) 2개 컬렉션 분리 운영
  - UUID5 기반 결정론적 ID로 중복 적재 방지
* **다중 지식베이스 Dense 검색 + Reranking**:
  - 두 컬렉션에서 각각 후보 추출 후 `BAAI/bge-reranker-v2-m3` CrossEncoder로 통합 재정렬
  - LLM Query Rewriting: Gemini API로 OCR 텍스트에서 핵심 법률 쟁점을 압축한 검색 쿼리 생성 (실패 시 키워드 기반 문장 추출 fallback)
* **검색 결과 후처리**:
  - `_summarize_cases`: 딱딱한 법률 문체의 검색 결과를 LLM으로 소비자 친화적 말투로 변환하여 프론트엔드에 전달

---

### ⚖️ 팀원 D (김동혁) : AI 룰 검증 및 QA 데이터 구축
> **주 사용 언어**: `Python`, `JSON`  
> **전담 작업 공간**: `backend/modules/rule_validator.py`, `backend/data/qa_dataset.json`
* **QA 데이터셋 설계 가이드 (1,000건 구축)**: 
  - `[페르소나 / 질문 / 모범답변 / 판정색상 / 적용법령]` 구조로 세분화하여 구축 (위약금, 허위광고 등 포괄)
* **수치 기반 교차 검증 (Cross-Validation) 가이드**:
  - 정규표현식(Regex)을 활용해 OCR 텍스트에서 '위약금 %' 등 핵심 숫자 직접 추출
  - 하드코딩된 법정 한도 상수와 비교하여 LLM 환각 발생 시 결과를 강제 교정(Override)하는 로직 구현
* **3색 신호등 라벨링(Labeling) 기준**:
  - 🔴 **RED (위험)**: 계약무효 및 명백한 강행법률 위반 사항 (예: 스포츠 시설 위약금 10% 한도 초과, 양도비 3만 원 초과, 7일 미만 청약철회 기간 제한, 단순 포장개봉 환불 거부, 부작용 없음 단정적 광고, 환불 절대 불가 약관 등). (관공서 신고서 노드 트리거)
  - 🟡 **YELLOW (주의)**: 계약 자체가 무효는 아니나 소비자의 심리를 왜곡하여 충동구매를 유도하는 마감 임박/선착순 등 다크패턴 기만 광고 및 소비자에게 일방적으로 불리한 모호한 조항.
  - 🟢 **GREEN (안전)**: 공정거래위원회 표준약관 및 관계 법령을 명확히 준수함.

## ⚠️ 팀원별 필수 유의사항 (협업 그라운드 룰)

프로젝트의 성공적인 완성을 위해 각 팀원은 아래 유의사항을 반드시 숙지하고 개발에 임합니다.

* 🌐 **[공통] Git 브랜치 전략 및 커밋 주의**: 
  - **절대 `main` 브랜치에 직접 커밋(Direct Push)하지 마세요.** `main`은 항상 에러 없이 실행되어야 하는 배포용 성역입니다.
  - 모든 팀원(리더인 팀원 A 포함)은 반드시 `main`에서 **자신의 기능용 브랜치(예: `feat/frontend`, `feat/rag` 등)를 파서** 그곳에 커밋해야 합니다.
  - 작업이 완료되면 **Pull Request(PR)**를 올리고, 백엔드 리더(팀원 A)가 전체 파이프라인이 깨지지 않는지 리뷰한 뒤에만 `main`에 병합(Merge)합니다.
  - `.env` 파일(API 키 등)은 절대로 Git에 올리지 않으며, 커밋 전 `git status`를 확인해 **자신에게 할당된 파일만 수정했는지 체크**하세요.
* 👑 **팀원 A (홍제희)**: 백엔드 API의 최종 출력(`AnalysisResponse`) 구조나 키(Key) 값을 부득이하게 변경해야 할 경우, **프론트엔드 담당인 팀원 B에게 즉시 공유**하여 화면이 터지는 것을 방지해야 합니다.
* 🎨 **팀원 B (김휘주)**: 백엔드 API가 아직 완성되지 않았더라도, 하드코딩된 임시 데이터(Mock Data)를 활용해 컴포넌트 렌더링이 정상적으로 작동하는지 **선행 테스트**를 진행해 주세요.
* 🔍 **팀원 C (이예경)**: Qdrant DB에 데이터를 적재(Ingest)할 때 사용한 **임베딩 모델**과, 실제 검색(`rag_search.py`)을 돌릴 때 사용하는 **임베딩 모델이 정확히 일치**해야만 유사도 검색이 작동합니다. 이 부분을 각별히 주의하세요.
* ⚖️ **팀원 D (김동혁)**: `qa_dataset.json` 파일을 수정할 때, 쉼표(`,`) 누락이나 쌍따옴표(`"`) 오류 등 **JSON 문법 에러**가 발생하지 않도록 주의하세요. 파이썬 정규식(Regex) 작성 시 다양한 예외 케이스(예: '30%', '삼십퍼센트')를 꼼꼼히 고려해야 합니다.
